# Production Deployment Runbook

Step-by-step instructions to run `mi` in production for one trusted community.
Read the **Operating model and limits** section at the end before going live —
it states what `mi` does and does not guarantee.

Architecture in one line: **one coordinator process per community** (the source
of truth), plus one **node-agent** on each provider machine (a Mac running
Ollama). Clients call the coordinator's OpenAI-compatible API.

Ready-to-use files referenced below:
- `configs/coordinator.prod.example.yaml`, `configs/node-agent.prod.example.yaml`
- `deploy/mi-coordinator.service`, `deploy/mi-backup.service`, `deploy/mi-backup.timer`
- `deploy/com.mi.node-agent.plist`

---

## 0. Decide before you start

- **Coordinator host**: one always-on Linux server (recommended) or an always-on Mac.
- **Node hosts**: the Apple Silicon Macs that will run inference (each needs Ollama).
- **Networking**: **Tailscale is the recommended path** — it gives an encrypted private network with no public ports, so you do not manage TLS certificates. (Alternative: expose the coordinator publicly behind TLS + node mTLS — see step 6B.)
- **Model**: pick one model every node will serve, e.g. `llama3.1:8b`.

You need Go 1.25+ on a build machine (any one host).

---

## 1. Build the binaries

On a build machine:

```bash
git clone https://github.com/raym33/mi.git
cd mi
make build        # produces bin/coordinator and bin/node-agent
```

Build for the target OS/arch if cross-compiling, e.g. for a Linux coordinator:

```bash
GOOS=linux GOARCH=amd64 go build -o bin/coordinator ./coordinator/cmd/coordinator
GOOS=linux GOARCH=amd64 go build -o bin/node-agent ./node-agent/cmd/node-agent
```

---

## 2. Private network (Tailscale, recommended)

Install Tailscale on the coordinator host and every node host, then:

```bash
sudo tailscale up
tailscale ip -4        # note the coordinator's 100.x.y.z address
```

Nodes and clients will reach the coordinator at `100.x.y.z:8080`. No public
exposure, traffic encrypted by WireGuard.

---

## 3. Coordinator host

```bash
sudo useradd --system --home /opt/mi --shell /usr/sbin/nologin mi || true
sudo mkdir -p /opt/mi/bin /etc/mi /var/lib/mi /var/backups/mi
sudo cp bin/coordinator /opt/mi/bin/
sudo cp -r scripts /opt/mi/scripts
sudo cp configs/coordinator.prod.example.yaml /etc/mi/coordinator.yaml
```

Edit `/etc/mi/coordinator.yaml`:

```bash
# generate a strong admin token and paste it into admin_token:
openssl rand -hex 32
```

- Set `admin_token` to that value. Keep `dev_admin_open: false`.
- Confirm `city.enabled: true` and `city.require_provider_tokens: true`.
- Make sure `challenges.model` and the `models.aliases[].target` match the model your nodes will serve.

Install and start the service:

```bash
sudo cp deploy/mi-coordinator.service /etc/systemd/system/
sudo chown -R mi:mi /opt/mi /var/lib/mi /var/backups/mi
sudo systemctl daemon-reload
sudo systemctl enable --now mi-coordinator
sudo systemctl status mi-coordinator --no-pager
curl -fsS http://localhost:8080/health      # -> {"status":"ok"}
```

---

## 4. Create accounts (admin API)

Secrets are generated by the coordinator and shown **once** — store them in your
password manager immediately. Set `ADMIN=` to your token.

```bash
ADMIN="<admin_token>"
BASE="http://localhost:8080"

# One provider per provider machine (or one shared). Capture provider_token:
curl -fsS -X POST "$BASE/admin/providers" -H "Authorization: Bearer $ADMIN" \
  -H 'Content-Type: application/json' \
  -d '{"id":"mac-studio-1","display_name":"Mac Studio 1","privacy_mode":"private"}'

# One consumer per client/team. Capture api_key:
curl -fsS -X POST "$BASE/admin/consumers" -H "Authorization: Bearer $ADMIN" \
  -H 'Content-Type: application/json' \
  -d '{"id":"team-a","display_name":"Team A","total_token_limit":2000000}'
```

Rotate or disable later with `POST /admin/{providers,consumers}/{id}/rotate-*`
and `DELETE /admin/{providers,consumers}/{id}` (or the admin dashboard at
`/admin/dashboard`).

---

## 5. Node hosts (each Mac with Ollama)

Install the backend and pull the model:

```bash
brew install ollama
ollama serve            # or run the Ollama app; keep it running
ollama pull llama3.1:8b
```

Install the node-agent:

```bash
sudo mkdir -p /usr/local/mi/bin /usr/local/mi/logs
sudo cp bin/node-agent /usr/local/mi/bin/
cp configs/node-agent.prod.example.yaml /usr/local/mi/node-agent.yaml
```

Edit `/usr/local/mi/node-agent.yaml`:
- `provider_id` and `provider_token` = the values from step 4.
- `coordinator_url` = `ws://100.x.y.z:8080/ws/node` (the coordinator's tailnet IP).
- `models` = `["llama3.1:8b"]` (must match what you pulled).

Run it always-on with launchd:

```bash
cp deploy/com.mi.node-agent.plist ~/Library/LaunchAgents/com.mi.node-agent.plist
launchctl load -w ~/Library/LaunchAgents/com.mi.node-agent.plist
tail -f /usr/local/mi/logs/node-agent.log    # should log "registered node ..."
```

Verify on the coordinator that the node is healthy and advertising the model:

```bash
curl -fsS "$BASE/network/status"
curl -fsS "$BASE/admin/nodes" -H "Authorization: Bearer $ADMIN"
```

---

## 6. Verify end to end

```bash
curl -fsS "$BASE/v1/chat/completions" \
  -H "Authorization: Bearer <consumer_api_key>" \
  -H 'Content-Type: application/json' \
  -d '{"model":"fast","messages":[{"role":"user","content":"hola"}],"stream":false}'
```

Then confirm accounting moved:

```bash
curl -fsS "$BASE/admin/settlement" -H "Authorization: Bearer $ADMIN"
curl -fsS "$BASE/admin/metrics"    -H "Authorization: Bearer $ADMIN" | grep mi_settlement
```

### 6B. Public exposure instead of Tailscale (optional)

If you must expose the coordinator on the internet, do NOT rely on plain `ws`.
Generate certificates, set the `tls:` block in the coordinator config
(`cert_file`, `key_file`, and `node_client_ca_file` to require node mTLS), set
each node's `coordinator_url` to `wss://your-host:8080/ws/node` and its `tls:`
block. See `docs/security.md`.

---

## 7. Operations

**Backups (do this — the hash chains detect tampering but cannot recover a deleted file):**

```bash
sudo cp deploy/mi-backup.service deploy/mi-backup.timer /etc/systemd/system/
sudo systemctl enable --now mi-backup.timer    # daily tarball of /var/lib/mi -> /var/backups/mi
```

**Tamper-evidence anchor** — publish periodically (email to members, a timestamping service):

```bash
ADMIN_TOKEN="<admin_token>" BASE_URL="$BASE" bash scripts/anchor-hash.sh
```

**Monitoring** — point Prometheus at the admin metrics (requires the bearer token):

```yaml
scrape_configs:
  - job_name: mi
    metrics_path: /admin/metrics
    authorization:
      credentials: "<admin_token>"
    static_configs:
      - targets: ["100.x.y.z:8080"]
```

**Upgrades** (zero-surprise, graceful):

```bash
# build new binary, then:
sudo systemctl stop mi-coordinator     # SIGTERM drains in-flight requests
sudo cp bin/coordinator /opt/mi/bin/
sudo systemctl start mi-coordinator
```

**Logs:** `journalctl -u mi-coordinator -f` (coordinator), `/usr/local/mi/logs/` (nodes).

## Operating model and limits (read before going live)

`mi` is a **local-first cooperative control plane for one trusted community**. It
is honest about its boundaries — make sure these match your expectations:

- **Single coordinator = single point of failure.** There is no built-in HA. The
  availability plan is active-passive: keep backups (step 7) and, for failover,
  restore `/var/lib/mi` onto a standby host and repoint nodes. Plan a short
  maintenance window for upgrades.
- **Privacy tiers are routing policy, not cryptography.** The node that runs a
  request sees the prompt in clear text. Keep sensitive work on `private`
  (trusted) nodes; this protects *which machine* sees data via trust + routing,
  not by encrypting it from the executor.
- **Settlement is cooperative accounting, not a payment rail.** It is a
  tamper-evident internal record for credits/invoices, not money movement.
- **Token metering is a coordinator-side estimate**, not exact tokenizer parity.
  Fine for cooperative accounting; do not bill real money off it as-is.
- **Idempotency keys** prevent double-charging on retries (send `Idempotency-Key`
  on non-streaming requests). Full cross-ledger crash-atomicity is still planned
  (`docs/atomicity-design.md`).
- **Only Ollama is a production backend today.** Other backends are roadmap.
- **Secrets**: set a strong `admin_token`, never enable `dev_admin_open`, and
  keep provider tokens / API keys out of git (create them via the admin API).

See `ARCHITECTURE.md` for the full trust model and failure semantics.
